## Purpose

Turbo UI — внутренняя Design System и Component Library агентства. Цель: стабильная, предсказуемая и масштабируемая система для быстрой production-разработки сайтов и приложений. Конституция задаёт обязательные правила разработки и расширения системы.

## Core Principles

### I. Single Source of Truth (tokens.json)

Вся визуальная и типографическая семантика определяется в `tokens.json`. Цвета, spacing, typography, radius и прочие design tokens MUST браться только из токенов. Никаких дублирующих источников (отдельные CSS-переменные, константы в коде) для одних и тех же решений.

### II. Запрет hardcoded значений

Запрещено: жёстко заданные цвета (hex/rgb в коде или CSS), произвольные отступы (magic numbers), произвольные размеры шрифтов, произвольные border-radius. Допустимо только обращение к токенам (например, `var(--color-*)`, `var(--spacing-*)`) или к API компонентов, которые сами используют токены.

### III. Component-first архитектура

Любая UI-логика реализуется через компоненты. Нет «разовых» div/button/span с кастомными стилями вместо использования компонентов из библиотеки. Новый UI-элемент — сначала кандидат в компонент библиотеки, а не локальный хак.

### IV. Только существующие компоненты

В production-коде используются исключительно компоненты из Turbo UI. Запрещено создавать одноразовые компоненты-заменители (кнопки, инпуты, карточки и т.д.) вместо расширения библиотеки. Если нужного компонента нет — сначала расширяют Turbo UI, затем используют его.

### V. Figma как reference, не источник кода

Figma — структурный и визуальный ориентир. Реализация в коде MUST следовать архитектуре Turbo UI (tokens, компоненты, API). Копирование стилей/разметки из Figma в обход токенов и компонентов запрещено.

### VI. Минималистичная API-архитектура

Публичный API компонентов и библиотеки — минимально необходимый. Избыточные props, дублирующие способы достижения одного результата, устаревшие API без deprecation path не допускаются. Новый API обосновывается необходимостью.

### VII. Чистота production-кода

В ветках и артефактах, идущих в production, не допускаются: отладочный код, закомментированные блоки «на потом», временные обходные пути без тикета на рефакторинг. Код готов к долгой поддержке.

### VIII. Контроль производительности

Изменения, влияющие на bundle size, runtime или рендер (новые зависимости, тяжёлые компоненты, глобальные стили), должны быть обоснованы. Регрессии по производительности считаются нарушением и откатываются или исправляются до мержа.

### IX. Расширение системы

Новые компоненты и токены вводятся по единым правилам: описание в документации/Storybook, использование токенов, соответствие принципам конституции. Breaking changes в публичном API следуют semantic versioning и описываются в changelog.

### X. Versioning (semantic versioning)

Версионирование — строго по semantic versioning (MAJOR.MINOR.PATCH). MAJOR — несовместимые изменения API; MINOR — новая функциональность без ломающих изменений; PATCH — исправления и обратно совместимые улучшения. Версия в `package.json` и тегах репозитория должна соответствовать реальным изменениям.

## Constraints

- Стек: React, Vite, Tailwind, Storybook. Введение новых ключевых технологий согласуется и документируется.
- Стили: только через design tokens и утилиты, опирающиеся на токены. Никаких произвольных значений в className/стилях для визуальных решений, определяемых дизайном.
- Тесты: критические пути и публичный API компонентов должны быть покрыты; регрессии при рефакторинге не допускаются без явного решения.

## Non-Goals

- Turbo UI не подменяет дизайн-процесс: дизайн остаётся в Figma/процессах агентства; библиотека — реализация и контракт.
- Не цель — максимальная гибкость «под любой дизайн»; приоритет — предсказуемость и согласованность в рамках Design System.
- Не цель — дублирование функциональности сторонних UI-библиотек; фокус на собственных компонентах и токенах.

## Enforcement

Нарушения конституции устраняются до мержа. Рекомендуемые действия при типичных нарушениях:

| Нарушение | Действие |
|-----------|----------|
| Hardcoded цвет/spacing/типографика | Заменить на ссылку на токен из `tokens.json` или переменную темы; при отсутствии токена — добавить токен, затем заменить значение. |
| Использование сырых HTML-элементов вместо компонентов | Обернуть в соответствующий компонент Turbo UI или добавить компонент в библиотеку и использовать его. |
| Одноразовый компонент вместо элемента библиотеки | Вынести в библиотеку (или расширить существующий компонент), использовать из библиотеки; удалить локальную копию. |
| Стили/разметка скопированы из Figma без токенов | Переписать с использованием токенов и компонентов; при необходимости дополнить набор токенов. |
| Раздутый или дублирующий API | Удалить избыточные props/варианты; оставить один явный способ; при breaking change — MAJOR bump и changelog. |
| Мусор в коде (debug, комментарии-заглушки, временные обходы) | Удалить перед мержем; временные обходы оформить тикетом и ограничить по времени. |
| Регрессия по bundle/runtime/рендеру | Откатить или оптимизировать до приёмки; зафиксировать в PR критерии (размер, метрики). |
| Несоответствие версии в package.json и характеру изменений | Привести версию в соответствие с semantic versioning; обновить changelog. |

Автоматизация: в CI рекомендуется проверки (например, линтеры/скрипты) на наличие типичных hardcoded значений (цвета, spacing) и использование токенов. Нарушения блокируют мерж.

## Governance

Конституция имеет приоритет над локальными соглашениями и «как мы всегда делали». Исключения допустимы только через поправку конституции с указанием причины и срока.

Поправки: документируются, версия конституции обновляется по semantic versioning (см. принцип X). Изменения принципов — MINOR или MAJOR; уточнения формулировок — PATCH. Даты: RATIFICATION_DATE — первое принятие; LAST_AMENDED_DATE — дата последней правки.

Все PR и ревью MUST проверять соответствие конституции. Вопросы трактуются в пользу конституции при отсутствии явного разрешённого исключения.
