# Feature Specification: Select и Listbox

**Feature Branch**: `009-select-listbox`  
**Created**: 2026-05-02  
**Status**: Draft  
**Input**: Два компонента дизайн-системы: **Select** — выбор одного значения из выпадающего списка (без лейбла и хелпера в этой итерации); **Listbox** — сам список опций (панель списка). Реализация по **конституции** Turbo UI: только токены проекта и макеты **Figma** как reference. Документация в **Storybook** — по шаблону библиотеки; тексты — коротко, с пользой для разработчиков и дизайнеров, без тавтологии (стиль «Пиши, сокращай»).

**Референсы Figma**

- **Select (триггер + поведение)**: [Turbo UI — node 830:590](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=830-590)
- **Listbox (контейнер списка)**: [Turbo UI — node 830:857](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=830-857)
- **Элементы списка (айтемы)**: [Turbo UI — node 830:683](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=830-683)

## Purpose

**Listbox** — переиспользуемая панель списка опций: скролл, состояния пунктов, опциональные иконки. **Select** — готовый паттерн «кнопка/триггер + список»: открытие/закрытие, выбор одного значения, позиционирование списка относительно триггера, поиск и фильтрация по запросу. Оба MUST визуально и по поведению согласовываться с макетами и с остальными контролями Turbo UI; оформление только через **design tokens** и принятую в репозитории модель стилей.

## Scope

- Публичные компоненты **Listbox** и **Select** с отдельными entry для потребителей; существующие публичные API не MUST ломаться.
- **Listbox**: отображение коллекции опций, поддержка варианта с **иконкой слева** (вкл/выкл по данным или настройке), индикация выбранного пункта **иконкой check справа** там, где сценарий (например Select) это задаёт; состояния пунктов и контейнера — по макетам **830:857** и **830:683**.
- **Select**: триггер без обвязки лейбл/хелпер; выпадающий **Listbox** (или его визуальный/семантический эквивалент в составе Select) с поведением выбора одного значения; состояния **disabled**, **ошибка (invalid)** — по макету **830:590**; поле поиска внутри списка при включённой опции — на базе существующего **Input** библиотеки.
- Позиционирование и геометрия списка относительно триггера — см. требования FR по **ширине**, **высоте списка**, **positions**, **menuOffset**, **placeholder**, **search**, **filterItem** (детали ниже).
- **Вне scope** этой спеки: составное поле с лейблом и текстом ошибки снаружи (как отдельный Field); мультивыбор; произвольные стили в обход токенов.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Select: выбрать одно значение из списка (P1)

Пользователь открывает список с триггера, видит опции, выбирает одну; триггер показывает выбранное значение или **placeholder**, если значения нет. При **disabled** открыть и сменить выбор нельзя. При **ошибке** триггер (и при необходимости контекст) читается как невалидный по макету.

**Why this priority**: основной сценарий продукта для выпадающего выбора.

**Independent Test**: Storybook или тестовый экран: открытие/закрытие, выбор, placeholder, disabled, invalid — без кастомной вёрстки обходными путями.

**Acceptance Scenarios**:

1. **Given** Select с опциями, **When** пользователь открывает список и выбирает пункт, **Then** список закрывается, в триггере отображается выбранная метка.
2. **Given** значение не задано и передан **placeholder**, **When** отображается триггер, **Then** виден текст placeholder.
3. **Given** **disabled**, **When** пользователь пытается открыть список или сменить значение, **Then** поведение и визуал соответствуют макету disabled.
4. **Given** режим **ошибки**, **When** отображается Select, **Then** визуал ошибки на триггере (и связанных элементах по макету) согласован с **830:590**.

---

### User Story 2 — Listbox как строительный блок (P1)

Разработчик вставляет **Listbox** (или его опции) в кастомный триггер или другой контейнер: пункты с текстом, опционально **иконка слева**, для выбранного — **check справа** в сценарии Select; скролл при переполнении по заданной высоте.

**Why this priority**: без стабильного списка Select не переиспользует единый визуал и поведение опций.

**Independent Test**: отдельная демонстрация Listbox с длинным набором опций и с вариантами с/без левой иконки и с отмеченным выбранным.

**Acceptance Scenarios**:

1. **Given** набор опций, **When** рендерится Listbox, **Then** пункты соответствуют макету **830:683** по типографике, отступам и состояниям.
2. **Given** для пункта включена левая иконка, **When** рендерится пункт, **Then** иконка видна слева от текста по макету.
3. **Given** сценарий «как в Select» для выбранного пункта, **When** пункт выбран, **Then** справа отображается отметка **check** по макету.

---

### User Story 3 — Позиция и границы экрана (P2)

Список позиционируется относительно триггера согласно переданным **positions** (одно значение или упорядоченный список). Если ни одна позиция из списка не укладывается во **viewport** без обрезки критичной части списка, используется **первая** позиция из переданного списка (зафиксированное правило приоритета). Горизонтальное смещение задаётся **menuOffset** (0 — без смещения; положительное — сдвиг списка **влево** относительно привязки; отрицательное — **вправо**).

**Why this priority**: предсказуемость на сложных экранах и в модалках.

**Independent Test**: смена `positions` и `menuOffset` в Storybook с визуальной проверкой краев экрана (в т.ч. искусственно узкий viewport).

**Acceptance Scenarios**:

1. **Given** передан один допустимый вариант **positions**, **When** список открыт, **Then** якорение соответствует этому варианту (top/bottom/left/right + left/center/right или top/middle/bottom — по выбранной комбинации из допустимого набора).
2. **Given** передан массив позиций и узкий viewport, **When** система не может разместить список ни в одной позиции без нарушения правила видимости, **Then** применяется **первая** позиция из массива.

---

### User Story 4 — Поиск и фильтрация (P2)

При включённой опции **search** над списком отображается строка поиска на компоненте **Input** библиотеки. Через **filterItem** потребитель задаёт правило отбора опций по введёному запросу (например подстрока, регистронезависимость — на усмотрение реализации в плане; контракт — «видимый поднабор опций»).

**Why this priority**: длинные списки в продуктах без внешнего поиска неудобны.

**Independent Test**: ввод в поле поиска сокращает видимый список согласно переданному **filterItem**; пустой запрос показывает полный набор (если иное не задано явно в контракте).

**Acceptance Scenarios**:

1. **Given** **search** включён, **When** открыт список, **Then** видна строка поиска, стилизованная через существующий **Input**.
2. **Given** **filterItem** задан, **When** пользователь вводит запрос, **Then** отображаются только опции, проходящие фильтр.

---

### User Story 5 — Документация Storybook (P2)

Страницы **Select** и **Listbox** в Storybook: матрицы состояний, позиции, поиск, иконки у айтемов; краткие тексты по шаблону проекта; ссылки на Figma.

**Why this priority**: снижает время интеграции и расхождения с дизайном.

**Independent Test**: открыть Docs, найти описание пропсов и сценариев без чтения исходников библиотеки.

**Acceptance Scenarios**:

1. **Given** запущенная документация, **When** открыты Select и Listbox, **Then** описаны назначение, связь компонентов, ключевые параметры и ограничения scope.

---

### Edge Cases

- **Нет опций** или **фильтр ничего не вернул**: пустое состояние списка не ломает фокус и закрытие; текст пустого состояния — в плане/копирайте (при необходимости минимальная заглушка).
- **Очень длинный список**: ограничение **высоты** списка обеспечивает скролл внутри панели, без выхода за пределы заданной высоты.
- **Ширина**: при заданных **фиксированной ширине** и **максимальной ширине** — правила пересечения (что доминирует) фиксируются в плане/контракте так, чтобы результат был однозначен.
- **Конфликт позиций и offset**: итоговая геометрия предсказуема; прижатие к краю экрана не обрезает критичный контент без явного правила в плане.
- **Ошибка без внешнего текста**: Select в invalid показывает визуал ошибки на триггере; связь с текстом ошибки формы — композиция продукта (как у других полей).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-1**: Система MUST предоставлять компонент **Select** для выбора **ровно одного** значения из списка опций; в этой фиче **без** обязательного лейбла и хелпера на уровне компонента.
- **FR-2**: Система MUST предоставлять компонент **Listbox** как список опций с визуалом и состояниями по макетам **830:857** и **830:683**.
- **FR-3**: Элемент списка MUST поддерживать опциональную **иконку слева** (вкл/выкл по данным пункта или общей политике — деталь в контракте); для сценария выбранного значения в Select MUST отображаться **иконка check справа** у выбранного пункта, как на макете.
- **FR-4**: Select MUST поддерживать задание **фиксированной ширины** и **максимальной ширины** (в терминах макета/токенов: без произвольных «сырых» значений там, где решение уже выражено токенами).
- **FR-5**: Select MUST поддерживать ограничение **высоты выпадающего списка** (прокручиваемая область опций).
- **FR-6**: Select MUST поддерживать проп **positions**: одно значение или список значений из набора: `top left`, `top center`, `top right`, `right top`, `right middle`, `right bottom`, `bottom left`, `bottom center`, `bottom right`, `left top`, `left middle`, `left bottom`. Порядок в списке задаёт приоритет перебора; если **ни одна** позиция из списка не удовлетворяет правилу укладки во **viewport**, MUST использоваться **первая** позиция из переданного списка.
- **FR-7**: Select MUST поддерживать проп **menuOffset**: при **0** смещения нет; при **положительном** значении список смещается **влево** относительно привязки к триггеру; при **отрицательном** — **вправо** (как во входном описании).
- **FR-8**: Select MUST поддерживать **placeholder** — текст при отсутствии выбранного значения.
- **FR-9**: При включённом **search** MUST отображаться строка поиска **на базе компонента Input** Turbo UI; MUST поддерживаться **filterItem** для отбора видимых опций по запросу.
- **FR-10**: Select MUST поддерживать **disabled** — блокировка открытия и смены значения, визуал по макету.
- **FR-11**: Select MUST поддерживать состояние **ошибки (invalid)** — визуал по макету **830:590**; поведение не должно вводить пользователя в заблуждение о доступности выбора.
- **FR-12**: Стили MUST задаваться только через токены проекта и принятую модель стилей; Figma — reference (конституция).
- **FR-13**: Публичный API MUST оставаться **минимально достаточным**; детальные имена пропов и типы — в контракте и плане.
- **FR-14**: Оба компонента MUST экспортироваться отдельными entry; документация в Storybook MUST следовать **шаблону** существующих компонентов; формулировки MUST быть ёмкими, без повторов одной мысли.

### Key Entities

- **Опция (item)**: отображаемая метка, значение, признак выбранности, доступность, опциональная левая иконка, опциональная правая отметка (check).
- **Список (Listbox)**: упорядоченная коллекция опций, область прокрутки, связь с клавиатурой и фокусом (уточнение в плане a11y).
- **Select**: триггер, открытое/закрытое состояние, текущее значение, параметры позиционирования и размеров панели, признаки disabled/invalid, опции поиска и фильтрации.

### Assumptions

- Правило «видимость во viewport» для FR-6 уточняется в плане измеримыми критериями (например допустимая обрезка тени vs контента), без противоречия зафиксированному приоритету «первая позиция при невозможности уложить».
- Точные токены для select/listbox при отсутствии в `tokens.json` добавляются по процессу дизайн-системы, а не разовыми константами в компонентах.
- Мультивыбор и нативный `<select>` как замена не входят в scope.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-1**: По чек-листу сопоставления с макетами **830:590**, **830:857**, **830:683** можно подтвердить визуальное соответствие триггера, панели списка и айтемов (включая иконки и check) для ключевых состояний без расхождений, влияющих на читаемость.
- **SC-2**: Разработчик может за **менее чем 30 минут** подключить Select с опциями, placeholder, одной позицией **positions** и **disabled** в тестовом проекте без правки кода библиотеки.
- **SC-3**: Для сценария с **search** и **filterItem** ввод запроса стабильно сокращает видимый список согласно заданному правилу в **100%** проверенных кейсов из плана приёмочных тестов.
- **SC-4**: Ревью по конституции: визуальные решения Select/Listbox не используют hardcoded значения там, где для них есть токены; отклонения устраняются до принятия фичи.
- **SC-5**: На страницах Docs Storybook для обоих компонентов читатель находит описание связи Listbox ↔ Select и перечисленных возможностей (позиции, offset, placeholder, поиск, ошибка) **без** обращения к исходникам.
