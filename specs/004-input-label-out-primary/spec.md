# Feature Specification: Input — вариант label-out-primary

**Feature Branch**: `004-input-label-out-primary`  
**Created**: 2026-04-09  
**Status**: Draft  
**Input**: На основе существующего Input реализовать вариант **label-out-primary** под публичным именем компонента **InputField** (модуль/entry `input-field`) по макетам Figma и токенам проекта; отразить в Storybook по аналогии с другими компонентами.

**Референс макета (Figma)**: [Turbo UI — node 696:412](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=696-412)

## Purpose

Вариант поля ввода с **внешней подписью** (label расположен **над** полем, не внутри рамки), опциональным текстом подсказки или ошибки **под** полем и визуалом **primary** (поверхность, обводки, типографика из токенов). Нужен для форм и экранов, где подпись должна быть читаемой отдельно от значения поля, в едином стиле Turbo UI.

## Scope

- Один продуктовый вариант **label-out-primary** поверх существующего компонента Input (расширение API и/или отдельный экспорт — на этапе плана): layout **колонка** — label → поле → helper/error.
- **Размеры**: large, medium, small (как в макете; отступы и типографика из токенов).
- **Состояния интерактива** (визуал как в макете): default, hover, focus, filled, filled-hover, disabled, invalid (ошибка).
- **Состав блока**: опциональный label; поле ввода с опциональной иконкой слева и опциональным IconButton справа (как у базового Input); опциональный helper; при invalid — сообщение об ошибке в зоне под полем (цвет и семантика по токенам ошибки).
- **Стили**: только токены темы и принятый в библиотеке способ стилизации компонентов (как у Input/Button); без «магических» чисел вне токенов.
- **Документация**: Storybook с примерами по размерам и состояниям, в том же подходе, что и для Input и других компонентов библиотеки.
- **Экспорт**: компонент **InputField** доступен потребителям библиотеки через entry `input-field` так же, как остальные публичные компоненты (tree-shakeable, типы).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик подключает InputField (label-out-primary) в проекте (P1)

Разработчик импортирует **InputField** из Turbo UI (`turbo-ui/input-field`), задаёт label, значение/placeholder, при необходимости helper, размер и состояние (в т.ч. ошибка и disabled). Поле ведёт себя как обычный Input (controlled/uncontrolled, фокус, ввод), подпись и подсказка связаны с полем для доступности.

**Why this priority**: Без использования в продукте фича не даёт ценности.

**Independent Test**: Тестовый проект или Storybook: рендер всех размеров с label и helper; смена состояния на error и disabled; ввод текста.

**Acceptance Scenarios**:

1. **Given** подключённый Turbo UI, **When** рендерится поле label-out-primary с текстом label и placeholder, **Then** label отображается **над** рамкой поля, placeholder — внутри поля.
2. **Given** поле в состоянии invalid, **When** пользователь видит интерфейс, **Then** обводка поля и текст под полем соответствуют состоянию ошибки в макете, у поля выставлены признаки доступности ошибки (как у базового Input).
3. **Given** поле disabled, **When** оно отображается, **Then** label, поле, иконки и helper визуально неактивны согласно макету, ввод невозможен.
4. **Given** передан helperText и нет ошибки, **When** отображается блок, **Then** под полем показывается нейтральная подсказка стилем caption/helper из токенов.

---

### User Story 2 — Визуальное соответствие макету и токенам (P1)

Реализация визуально совпадает с макетом Figma (node выше): отступы между label, полем и helper (например, вертикальный шаг из токенов spacing), скругления, обводки default/hover/focus/invalid, типографика label / текст поля / helper по шкалам typescale из темы.

**Why this priority**: Иначе вариант не является частью дизайн-системы.

**Independent Test**: Сравнение скриншота Storybook (или статичных состояний) со скриншотом макета по всем размерам и перечисленным состояниям.

**Acceptance Scenarios**:

1. **Given** макет и собранный вариант в теме по умолчанию, **When** сравниваются default, hover, focus, filled, filled-hover, disabled, invalid для large, medium, small, **Then** нет систематических расхождений по отступам, размерам иконок и иерархии текста относительно макета.
2. **Given** смена темы на поддерживаемую библиотекой, **When** отображается компонент, **Then** цвета и обводки берутся из токенов (content, border, surface), а не из фиксированных значений вне системы.

---

### User Story 3 — Документация в Storybook (P2)

В Storybook есть страница (или секция) для **InputField** (вариант label-out-primary): матрица размеров × состояний, при необходимости отдельные истории с и без иконок, с helper и с ошибкой — по аналогии с документацией Input и других компонентов.

**Why this priority**: Потребители библиотеки должны находить API и примеры без обращения к исходникам.

**Independent Test**: Открыть Storybook, убедиться в наличии описанных примеров и соответствии названий пропсов/вариантов контракту.

**Acceptance Scenarios**:

1. **Given** репозиторий Turbo UI, **When** запускается Storybook, **Then** доступна документация варианта label-out-primary с перечисленными комбинациями.
2. **Given** контракт компонента, **When** разработчик читает Storybook/описание пропсов, **Then** ясно, как включить label-out layout и primary-стиль относительно базового Input.

---

### Edge Cases

- Длинный label или длинный helper: перенос или обрезка по правилам дизайн-системы; блок не ломает вёрстку соседних полей.
- Одновременно helper и ошибка: при ошибке показывается сообщение об ошибке в зоне под полем (как в макете invalid — красный текст); приоритет ошибки над нейтральным helper.
- Отсутствие label или helper: блок сжимается по вертикали без лишних пустых зон (или с зарезервированным местом — если базовый Input уже задаёт правило «без прыжков»; наследовать поведение базового компонента).
- Навигация с клавиатуры и screen reader: наследовать требования базового Input (связь label–поле, aria-invalid, aria-describedby).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-1**: Вариант MUST отображать опциональный **label над** полем ввода (не внутри границы поля), с типографикой уровня label/small из токенов (как в макете).
- **FR-2**: Вариант MUST поддерживать размеры **large**, **medium**, **small**; высота поля, размеры иконок и кегль текста в поле MUST соответствовать макету для каждого размера.
- **FR-3**: Вариант MUST поддерживать визуальные состояния: default, hover, focus, filled, filled-hover, disabled, invalid; границы и цвет текста MUST опираться на токены (border tertiary/secondary, border error, content primary/tertiary/disabled/error и т.д. по макету).
- **FR-4**: Поле ввода MUST переиспользовать поведение и ограничения базового Input (типы поля, IconButton справа, иконка слева, controlled/uncontrolled, forwardRef, расширение нативных атрибутов) — конкретное API фиксируется в плане/контракте.
- **FR-5**: Опциональный текст под полем MUST использовать стиль caption/helper из токенов; в состоянии invalid текст под полем MUST быть в цвете ошибки (как в макете).
- **FR-6**: Layout MUST сохранять порядок: label (если есть) → поле → helper/error (если есть); вертикальные отступы между блоками MUST соответствовать макету через токены spacing.
- **FR-7**: Компонент MUST экспортироваться из библиотеки для внешних потребителей; существующие публичные API других компонентов не MUST ломаться.
- **FR-8**: Документация в Storybook MUST включать представление варианта по размерам и ключевым состояниям, согласованно с остальными компонентами Turbo UI.

### Assumptions

- Базовый компонент Input из спеки `003-input` уже существует или будет доступен к моменту реализации; label-out-primary — его вариант по layout/стилю, а не отдельная несвязанная реализация поля.
- Иконка слева и IconButton справа — те же примитивы, что у Input; размеры кнопки-иконки привязаны к размеру поля как в макете (large/medium/small).
- «Primary» в названии варианта соответствует набору токенов surface/border/content для основного светлого поля на макете; альтернативные темы подстраиваются через те же семантические токены.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-1**: Для каждого размера (large, medium, small) и каждого состояния (default, hover, focus, filled, filled-hover, disabled, invalid) визуальное представление можно сопоставить с макетом Figma без расхождений, влияющих на читаемость и иерархию (проверка чек-листом визуальной регрессии или ручным сравнением со скриншотом макета).
- **SC-2**: Разработчик может за менее чем 15 минут подключить вариант в тестовом проекте и воспроизвести сценарии: с label и helper, с ошибкой, disabled — без правки кода библиотеки.
- **SC-3**: В Storybook доступны примеры, покрывающие все размеры и основные состояния варианта label-out-primary.
- **SC-4**: Для сценария ошибки пользователь ассистивных технологий получает объявление об ошибке и связь текста ошибки с полем (критерии наследуются от базового Input и не регрессируют).
