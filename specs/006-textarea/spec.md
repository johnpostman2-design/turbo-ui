# Feature Specification: TextArea

**Feature Branch**: `006-textarea`  
**Created**: 2026-04-17  
**Status**: Draft  
**Input**: Создать новый компонент дизайн-системы — **TextArea**: многострочное поле ввода со стандартными состояниями (дефолт, ховер, фокус, ошибка, дизейбл) и визуалом по макету Figma; соответствие конституции Turbo UI; согласованность с остальными компонентами полей ввода; только стили из токенов проекта; простой API для разработчиков.

**Референс макета (Figma)**: [Turbo UI — node 726:146](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=726-146)

## Purpose

**TextArea** — компонент для ввода и отображения текста в **несколько строк** (в отличие от однострочного Input). Нужен для форм, комментариев, описаний и других сценариев, где объём текста не умещается в одну строку. Визуальная и поведенческая модель должна быть **согласована** с остальными полями Turbo UI (типографика, отступы, обводки, семантика ошибки и disabled), а реализация — опираться **только на design tokens** и принципы конституции (без произвольных «магических» значений вне системы).

## Scope

- Один публичный компонент **TextArea** для многострочного ввода на базе нативной семантики многострочного поля (высокоуровневые детали API — на этапе плана и контракта).
- **Размеры**: **medium** и **small**, как на макете; высота поля и типографика текста внутри поля соответствуют макету для каждого размера. При необходимости выравнивания с линейкой Input допускается отдельное решение на этапе плана (например, добавление **large**), если это не противоречит макету и токенам.
- **Состояния взаимодействия и отображения** (визуал как в макете и в запросе): **default**, **hover**, **focus**, состояние с **введённым значением** (filled), **disabled**, **ошибка (invalid)** с текстом ошибки **под** полем.
- **Вариант типа**: **primary** (основное поле на поверхности из токенов), в духе макета.
- **Ошибка**: обводка в семантике ошибки; под полем — сообщение об ошибке стилем caption/error из токенов (как на макете для invalid).
- **Индикатор изменения размера**: на макете присутствует ручка resize в углу поля; отображение и допустимое поведение изменения размера (например, только вертикально) задаются по макету и правилам доступности, без дублирования произвольных цветов из инструментов экспорта.
- **Стили**: только токены темы (`tokens.json` / CSS-переменные темы) и принятый в библиотеке способ стилизации компонентов; Figma — reference, не источник «сырых» hex в коде.
- **Документация**: Storybook с матрицей размеров × состояний и примерами использования — **в том же подходе**, что и для Input и других компонентов (шаблон документации библиотеки).
- **Экспорт**: компонент доступен потребителям библиотеки через отдельный entry (как у других UI-компонентов), с типами и tree-shakeable экспортом.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик подключает TextArea в продукте (P1)

Разработчик импортирует TextArea из Turbo UI, задаёт значение и/или placeholder, при необходимости — размер, состояние ошибки и `disabled`. Поле поддерживает многострочный ввод, прокрутку содержимого при длинном тексте и согласованное с другими полями поведение фокуса и подсказок для доступности.

**Why this priority**: Без использования в реальном экране компонент не даёт ценности.

**Independent Test**: Тестовый проект или Storybook: рендер medium и small; ввод и редактирование текста; переключение на ошибку и disabled; проверка связи подписи/ошибки с полем там, где API это предусматривает.

**Acceptance Scenarios**:

1. **Given** подключённый пакет Turbo UI, **When** рендерится TextArea с placeholder, **Then** отображается многострочная область с placeholder в соответствии с размером и макетом.
2. **Given** пользователь вводит несколько строк текста, **When** содержимое длиннее видимой области, **Then** поле ведёт себя ожидаемо для многострочного ввода (прокрутка или рост по правилам реализации, согласованным с макетом).
3. **Given** передан признак ошибки и текст ошибки, **When** отображается компонент, **Then** видна обводка ошибки и сообщение под полем в стиле ошибки из токенов; у поля выставлены признаки доступности для ошибки (согласованно с подходом Input).
4. **Given** `disabled`, **When** пользователь взаимодействует с полем, **Then** ввод невозможен, визуал соответствует состоянию disabled на макете.

---

### User Story 2 — Визуальное соответствие макету и конституции (P1)

Реализация для каждого размера и состояния визуально согласована с макетом Figma (node выше): обводки default/hover/focus/disabled/invalid, фон, типографика placeholder и значения, отступы внутри поля, скругления, строка ошибки под полем для invalid. Цвета и отступы не задаются произвольными константами вне токенов.

**Why this priority**: Иначе компонент не является частью единой дизайн-системы.

**Independent Test**: Сравнение статичных состояний в Storybook со скриншотом макета по строкам сетки (размер × состояние).

**Acceptance Scenarios**:

1. **Given** тема по умолчанию библиотеки, **When** сравниваются default, hover, focus, filled, disabled, invalid для medium и small, **Then** нет систематических расхождений по обводкам, контрасту текста и положению текста ошибки относительно макета.
2. **Given** проверка на соответствие конституции, **When** ревью визуальных решений, **Then** для границ, фона и текста используются семантические токены темы, а не дублирующиеся источники.

---

### User Story 3 — Документация в Storybook (P2)

В Storybook есть страница (или группа историй) для TextArea: матрица размеров и состояний, пример с ошибкой, пример disabled, краткое описание пропсов — по образцу существующих компонентов библиотеки.

**Why this priority**: Разработчики должны находить API и примеры без чтения исходников.

**Independent Test**: Открыть Storybook и убедиться в наличии описанных примеров и понятных названий вариантов.

**Acceptance Scenarios**:

1. **Given** репозиторий Turbo UI, **When** запускается Storybook, **Then** доступна документация TextArea с перечисленными комбинациями.
2. **Given** читатель документации, **When** изучает страницу TextArea, **Then** понятно, как задать размер, ошибку, disabled и базовое значение/placeholder.

---

### Edge Cases

- Очень длинный текст: поле остаётся используемым (прокрутка или политика роста — по решению плана, не нарушая макет и доступность).
- Пустое значение vs placeholder: отображение placeholder в default, без подмены семантики значения.
- Ошибка без текста сообщения: при необходимости уточняется в плане; при отсутствии текста ошибки визуал и a11y не должны вводить пользователя в заблуждение (минимум — не показывать пустую строку ошибки как обязательную).
- Изменение размера пользователем: если включено, не ломает вёрстку формы; курсор/ручка соответствуют макету.
- Согласование с формами: поведение рядом с Input и InputField — единые отступы и типографика там, где компоненты стоят в одной форме (по токенам).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-1**: Система MUST предоставлять компонент **TextArea** для многострочного ввода текста с отображением содержимого на нескольких строках.
- **FR-2**: TextArea MUST поддерживать размеры **medium** и **small** в соответствии с макетом; высота и внутренние отступы MUST опираться на токены.
- **FR-3**: TextArea MUST поддерживать визуальные и интерактивные состояния: default, hover, focus, отображение введённого значения (filled), disabled и invalid; границы и цвет текста MUST соответствовать макету через семантические токены (включая border/error, content/disabled и т.д.).
- **FR-4**: В состоянии **invalid** под полем MUST отображаться сообщение об ошибке в стиле caption/error из темы; обводка поля MUST отражать ошибку.
- **FR-5**: Стили MUST задаваться только через токены проекта и принятую в библиотеке модель стилей; запрещены дублирующие «сырые» значения для тех же решений, которые уже выражены токенами (согласно конституции).
- **FR-6**: Публичный API MUST оставаться **минимально достаточным** для описанных сценариев (без дублирующих способов задать одно и то же); детализация пропсов — в плане и контракте.
- **FR-7**: TextArea MUST быть **согласован** с остальными компонентами полей ввода Turbo UI по семантике состояний (ошибка, disabled, фокус) и по использованию темы.
- **FR-8**: Компонент MUST экспортироваться для внешних потребителей библиотеки отдельным entry; существующие публичные API других компонентов не MUST ломаться.
- **FR-9**: Документация в Storybook MUST включать представление размеров и ключевых состояний, согласованно с остальными компонентами Turbo UI.
- **FR-10**: Индикатор и поведение **resize** MUST соответствовать макету (наличие ручки) и не противоречить выбранной политике доступности и вёрстки форм.

### Assumptions

- Токены для границ, поверхностей, текста и типографики уже покрывают сценарии primary-поля; при обнаружении пробела на этапе реализации токен дополняется по процессу дизайн-системы, а не через разовые константы в компоненте.
- Поведение **filled** визуально различает введённый текст от placeholder; детали совпадения с отдельной строкой «Filled» на макете сверяются при приёмке по скриншоту.
- Состав «подпись над полем» как отдельного варианта (аналог InputField) не обязателен в этой фиче, если макет node 726:146 задаёт только само поле и ошибку под ним; композиция с внешним label остаётся на стороне потребителя или последующих вариантов.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-1**: Для каждого размера (**medium**, **small**) и каждого согласованного с макетом состояния (default, hover, focus, filled, disabled, invalid) визуальное представление можно проверить по чек-листу сопоставления со скриншотом макета без расхождений, влияющих на читаемость и иерархию.
- **SC-2**: Разработчик может за **менее чем 15 минут** подключить TextArea в тестовом проекте и воспроизвести сценарии: ввод многострочного текста, ошибка с текстом под полем, disabled — без изменения кода библиотеки.
- **SC-3**: В Storybook доступны примеры, покрывающие оба размера и перечисленные ключевые состояния.
- **SC-4**: Ревью по конституции: для визуальных решений компонента не используются hardcoded значения там, где для них есть токены; нарушения устраняются до принятия фичи.
