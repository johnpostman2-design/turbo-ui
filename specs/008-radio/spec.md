# Feature Specification: Radio (радиокнопка)

**Feature Branch**: `008-radio`  
**Created**: 2026-05-02  
**Status**: Draft  
**Input**: Новый компонент дизайн-системы **Radio**: выбор **одного** значения из нескольких взаимоисключающих вариантов. Реализация по **конституции** Turbo UI: только токены и принятые в проекте стили, без произвольных «сырых» значений. Компонент должен быть рабочим, простым в подключении и описанным в **Storybook** по тому же шаблону документации, что у остальных компонентов библиотеки. Размеры: **large**, **medium**, **small**. У интерактивных вариантов предусмотрен **hover** во всех сочетаниях, **кроме** вариантов **disabled** (в disabled-состоянии hover не меняет внешний вид).

**Референс макета (Figma)**: [Turbo UI — node 794:352](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=794-352)

## Purpose

**Radio** — контроль для выбора одного варианта из набора (в паре с группой или общим контекстом значения). Нужен в формах, настройках, опросах и везде, где недопустим одновременный выбор нескольких опций из одной группы. Визуал и поведение MUST согласовываться с макетом и с остальными контролами Turbo UI; оформление только через **design tokens** и правила конституции.

## Scope

- Публичный компонент **Radio** с семантикой «один выбранный пункт в группе»: отображение состояния **выбран / не выбран**, поддержка **disabled** и **invalid** по макету; клавиатура и вспомогательные технологии — ожидаемое поведение переключателя выбора (детали доступности — в плане).
- **Размеры**: **large**, **medium**, **small** — как на макете; масштабируются индикатор (круг), зазор до подписи и стиль подписи в соответствии с типографикой размера.
- **Состояния отображения и взаимодействия** (по макету): **default**, **hover** (для всех интерактивных комбинаций **кроме disabled**), **focus** (видимый контур фокуса в духе системы), **disabled** (без hover-изменения внешнего вида), **invalid** (ошибка поля или группы — визуал invalid на макете, в том числе при hover там, где макет это показывает).
- **Значения выбора**: только **включён (выбран)** и **не выбран** — без третьего «неопределённого» состояния (в отличие от Checkbox).
- **Группа «одно из нескольких»**: спека задаёт внешний вид и поведение **одной** радиокнопки; связь нескольких вариантов с одним значением (группа, имя, навигация стрелками) может быть **композицией** на стороне продукта или отдельным составным паттерном в плане — без расширения scope на целую форму.
- **Стили**: только токены темы и принятая в репозитории модель стилизации; Figma — reference, не источник дублирующих вне системы значений в коде.
- **Документация**: Storybook — матрица размеров × состояний (включая hover там, где применимо, и отсутствие hover у disabled), примеры invalid и disabled; страница Docs по **тому же шаблону**, что у Button / Checkbox / TextArea (навигация, ссылка на Figma при необходимости, блоки с примерами). Тексты — кратко, по делу: назначение, как задаётся выбор и ошибка, что смотреть дизайнеру.
- **Экспорт**: отдельный entry для потребителей библиотеки, без ломки существующих публичных API.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик подключает Radio и получает выбор одного значения (P1)

Разработчик использует один или несколько Radio с общим контекстом значения так, что в группе может быть выбран не более одного варианта. Задаётся размер, выбранность, при необходимости — disabled и invalid. Активация мышью и с клавиатуры приводит к ожидаемому переключению выбора; disabled блокирует смену.

**Why this priority**: Без сценария «один из нескольких» радиокнопка не выполняет свою роль в интерфейсе.

**Independent Test**: Storybook или тестовый экран: несколько Radio с одним значением; выбор одного снимает выбор с другого; disabled не меняется при активации; invalid виден по макету.

**Acceptance Scenarios**:

1. **Given** подключённый пакет Turbo UI, **When** рендерится Radio в состоянии «не выбран», **Then** отображается визуал по размеру и макету.
2. **Given** два варианта в одной логической группе, **When** пользователь выбирает второй, **Then** первый становится невыбранным (поведение «одно из нескольких» воспроизводится без обходных визуальных хаков).
3. **Given** **disabled**, **When** пользователь пытается изменить выбор, **Then** значение не меняется, визуал disabled по макету; **при наведении** внешний вид **остаётся** как у disabled (без hover-варианта).
4. **Given** **invalid**, **When** отображается компонент, **Then** визуал ошибки соответствует макету; для интерактивных вариантов invalid hover отличается от default так, как на макете.

---

### User Story 2 — Визуальное соответствие макету и конституции (P1)

Для каждой комбинации размера, состояния (default, hover где применимо, focus, disabled, invalid) и выбранности (да/нет) внешний вид согласован с Figma (node выше). Цвета, обводки, фокус и отступы выражены через семантические токены.

**Why this priority**: Иначе компонент ломает единообразие дизайн-системы.

**Independent Test**: Сравнение статичных кадров в Storybook со скриншотом макета по матрице (размер × состояние × выбран/не выбран).

**Acceptance Scenarios**:

1. **Given** тема по умолчанию, **When** просматриваются large / medium / small для ключевых состояний, **Then** нет систематических расхождений с макетом по контрасту, обводкам и размеру индикатора.
2. **Given** ревью на соответствие конституции, **When** проверяются стили, **Then** нет произвольных дублирующих значений там, где решение уже выражено токенами.

---

### User Story 3 — Документация в Storybook (P2)

В Storybook доступна страница Radio: матрица примеров, invalid и disabled, краткое описание сценариев использования и параметров — в том же подходе, что у других компонентов; формулировки короткие и практичные.

**Why this priority**: Снижает время на интеграцию и согласование с дизайном.

**Independent Test**: Открыть Docs Radio и убедиться в наличии перечисленного и понятных заголовков секций.

**Acceptance Scenarios**:

1. **Given** запущенный Storybook, **When** открыта документация Radio, **Then** видны размеры, состояния, выбран/не выбран, примеры ошибки и блокировки.
2. **Given** читатель документации, **When** читает описание, **Then** понятно, как задать размер, выбранность, disabled и invalid без чтения исходников.

---

### Edge Cases

- **Только один выбранный в группе**: при необходимости управления с сервера или из формы — граничные случаи «ни один не выбран» и «два выбраны» обрабатываются на стороне продукта; компонент отображает переданное состояние выбранности честно.
- **Invalid** без текста ошибки рядом: визуал invalid не заменяет понятное сообщение об ошибке; связь с текстом — через композицию формы (как у других полей).
- **Фокус**: фокус виден там, где это задано макетом и системой; переход фокуса между пунктами группы не должен ломать ожидания пользователей клавиатуры (уточнение в плане a11y).
- Подпись к Radio: в этой фиче может быть **частью композиции** с типографикой темы или встроенной подписью — уточнение в плане по соответствию макету «индикатор + Text».

## Requirements *(mandatory)*

### Functional Requirements

- **FR-1**: Система MUST предоставлять компонент **Radio** для взаимоисключающего выбора: визуально и по смыслу один вариант «включён» из логической группы.
- **FR-2**: Radio MUST поддерживать размеры **large**, **medium** и **small** в соответствии с макетом; геометрия индикатора, отступ до подписи и стиль подписи MUST опираться на токены.
- **FR-3**: Radio MUST поддерживать состояния **default**, **hover** (для всех интерактивных сочетаний **кроме disabled**), **focus**, **disabled** и **invalid**; для **disabled** hover MUST **не** менять внешний вид (как зафиксировано во входном описании и на макете).
- **FR-4**: Для состояний **выбран** и **не выбран** MUST быть различимый визуал по макету.
- **FR-5**: Стили MUST задаваться только через токены проекта и принятую модель стилей библиотеки; запрещены дублирующие сырые значения для решений, уже выраженных токенами (конституция).
- **FR-6**: Публичный API MUST оставаться **минимально достаточным**; детали пропсов — в плане и контракте.
- **FR-7**: Radio MUST быть **согласован** с остальными контролами Turbo UI по фокусу, disabled и invalid там, где это применимо.
- **FR-8**: Компонент MUST экспортироваться отдельным entry; существующие публичные API других компонентов не MUST ломаться.
- **FR-9**: Документация в Storybook MUST включать представление размеров, состояний (включая отсутствие hover у disabled) и выбран/не выбран; тексты MUST быть краткими и полезными для разработчиков и дизайнеров.
- **FR-10**: Поведение **disabled** и **invalid** MUST соответствовать макету и не вводить пользователя в заблуждение относительно доступности действия.

### Key Entities

- **Вариант выбора (Radio)**: один пункт; атрибуты: принадлежность к группе (логически), выбран или нет, доступность для изменения, признак ошибки, размер.
- **Группа вариантов**: набор взаимоисключающих пунктов с одним текущим значением; в спеке как контекст использования, без детализации реализации группы.

### Assumptions

- При нехватке токенов для радиокнопки токены дополняются по процессу дизайн-системы, а не разовыми константами в компоненте.
- Составной компонент уровня «поле формы с лейблом и хинтом» не обязателен в этой фиче, если макет задаёт строку «индикатор + подпись» — допускается композиция; единый визуальный ряд с макетом — в плане.
- Навигация стрелками по группе и роль `radiogroup` при необходимости — в плане (a11y), без расширения функциональной спеки на уровень целой библиотеки форм.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-1**: Для каждого размера (**large**, **medium**, **small**) и согласованных с макетом состояний (default, hover где не disabled, focus, disabled, invalid) и выбранности (да/нет) визуальное соответствие можно проверить по чек-листу сопоставления со скриншотом макета без расхождений, влияющих на читаемость и иерархию.
- **SC-2**: Разработчик может за **менее чем 15 минут** подключить Radio в тестовом проекте и воспроизвести сценарии: выбор одного из нескольких, disabled и invalid — без правки кода библиотеки.
- **SC-3**: В Storybook доступны примеры, покрывающие три размера, ключевые состояния и выбран/не выбран; описание на странице Docs позволяет понять назначение и основные параметры без чтения реализации.
- **SC-4**: Ревью по конституции: для визуальных решений компонента не используются hardcoded значения там, где для них есть токены; нарушения устраняются до принятия фичи.
