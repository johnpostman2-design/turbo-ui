# Feature Specification: Turbo UI (Root)

**Feature Branch**: `001-turbo-ui-root-spec`  
**Created**: 2025-03-03  
**Status**: Draft  
**Input**: Root specification и конституция Design System Turbo UI. Язык — русский, технические термины — английский.

## Purpose

Turbo UI — внутренняя Design System и Component Library агентства. Библиотека компонентов синхронизирована с одноимённой библиотекой в Figma. Цель: в каждом новом проекте менять стили и tokens без изменения функционала библиотеки. Система используется для ускоренной production-разработки сайтов и внутренних приложений и должна обеспечивать стабильность, повторяемость и архитектурную дисциплину. Компоненты и стили тянутся из Figma-файла; компоненты хранятся в репозитории, описание отображается в Storybook. Компоненты подключаются в любой проект.

Документ служит root specification и project constitution для Spec Kit.

## Scope

Все принципы, ограничения и правила Enforcement в этом документе относятся **только к компонентам Turbo UI**: к коду самой библиотеки и к использованию её компонентов в проектах. В проектах допускаются блоки и вёрстка по кастомным дизайнам — отдельные секции, лейауты, уникальные UI-элементы, сверстанные под конкретный проект и не входящие в Turbo UI. Такие блоки не обязаны следовать данной спецификации. Всё, что касается компонентов из Turbo UI (их реализация в библиотеке, подключаемые токены, использование в проектах), MUST следовать этой спецификации.

## Core Principles

### I. Single Source of Truth (tokens.json)

Вся визуальная и типографическая семантика определяется в `tokens.json`. Цвета, spacing, typography, radius — только из токенов.

### II. Запрет hardcoded значений

Запрещены жёстко заданные цвета (hex/rgb в коде или CSS), произвольные отступы, размеры шрифтов, border-radius. Допустимо только обращение к tokens или к API компонентов, использующих токены.

### III. Component-first архитектура (в рамках Turbo UI)

Вся UI-логика, реализуемая компонентами Turbo UI, идёт через компоненты библиотеки. В коде Turbo UI нет «разовых» div/button/span с кастомными стилями вместо компонентов библиотеки.

### IV. Только существующие components (при использовании Turbo UI)

Когда в проекте используются компоненты Turbo UI, применяют только существующие компоненты библиотеки. Запрещено создавать одноразовые компоненты-заменители вместо компонентов Turbo UI. Если нужного компонента нет — сначала расширяют Turbo UI, затем используют его. Блоки проекта, не относящиеся к Turbo UI (кастомная вёрстка под дизайн проекта), могут быть сверстаны отдельно.

### V. Figma как reference, не источник кода

Figma — структурный и визуальный ориентир. Реализация в коде следует архитектуре Turbo UI (tokens, компоненты, API). Копирование стилей/разметки из Figma в обход токенов и компонентов запрещено.

### VI. Минималистичная API-архитектура

Публичный API компонентов и библиотеки — минимально необходимый. Избыточные props и дублирующие способы достижения одного результата не допускаются.

### VII. Отсутствие fallback-хаоса

Единая, предсказуемая стратегия fallback для состояний (loading, error, empty). Нет произвольных или противоречащих друг другу fallback в разных компонентах.

### VIII. Контроль performance

Изменения, влияющие на bundle size, runtime или рендер, должны быть обоснованы. Регрессии по производительности откатываются или исправляются до мержа.

### IX. Чистота production-кода

В production не допускаются отладочный код, закомментированные блоки «на потом», временные обходные пути без тикета на рефакторинг.

### X. Versioning (semantic versioning)

Версионирование по semantic versioning (MAJOR.MINOR.PATCH). Версия в `package.json` и тегах соответствует реальным изменениям.

### XI. Правила расширения Design System

Новые компоненты и токены вводятся по единым правилам: описание в документации/Storybook, использование токенов, соответствие принципам конституции. Breaking changes в публичном API следуют semantic versioning и описываются в changelog.

## Constraints

- Стили — только через design tokens и утилиты на их основе. Нет произвольных значений в className/стилях для решений, определяемых дизайном.
- Критические пути и публичный API компонентов должны быть покрыты тестами.

## Non-Goals

- Turbo UI не подменяет дизайн-процесс; библиотека — реализация и контракт.
- Не цель — максимальная гибкость «под любой дизайн»; приоритет — предсказуемость и согласованность.
- Не цель — дублирование функциональности сторонних UI-библиотек.

## Enforcement

Применимо только к компонентам Turbo UI (код библиотеки и использование её компонентов). Нарушение правил требует рефакторинга до мержа.

| Нарушение | Действие |
|-----------|----------|
| Hardcoded цвет, spacing, типографика, radius (в компонентах Turbo UI) | Заменить на ссылку на token из `tokens.json` или переменную темы; при отсутствии токена — добавить токен, затем заменить значение. |
| Raw HTML (div/button/span с кастомными стилями) вместо компонентов Turbo UI | Заменить на соответствующий компонент Turbo UI или добавить компонент в библиотеку и использовать его. |
| Inline styles (в компонентах Turbo UI) | Удалить; перенести стили в tokens + компоненты или утилиты на основе токенов. Inline styles в Turbo UI запрещены. |
| Одноразовый компонент вместо элемента библиотеки Turbo UI | Вынести в Turbo UI (или расширить существующий компонент), использовать из библиотеки; удалить локальную копию. |
| Стили/разметка в Turbo UI скопированы из Figma без токенов | Переписать с использованием токенов и компонентов. |
| Раздутый или дублирующий API | Удалить избыточные props/варианты; при breaking change — MAJOR bump и changelog. |
| Регрессия по bundle/runtime/рендеру | Откатить или оптимизировать до приёмки. |

В CI Turbo UI рекомендуется проверка на hardcoded значения, использование токенов и отсутствие inline styles. Нарушения блокируют мерж. Кастомные блоки в проектах (не из Turbo UI) не подпадают под эти проверки.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик подключает Turbo UI в новый проект (Priority: P1)

Разработчик добавляет Turbo UI как зависимость, настраивает тему (tokens) под проект и использует компоненты без изменения кода библиотеки.

**Why this priority**: Без возможности подключения в любой проект библиотека не выполняет свою цель.

**Independent Test**: Установка пакета в чистом проекте, переопределение токенов, рендер нескольких компонентов; проверка, что стили применяются из токенов.

**Acceptance Scenarios**:

1. **Given** новый проект, **When** добавляют зависимость Turbo UI и переопределяют tokens, **Then** компоненты отображаются с новыми стилями без правки кода библиотеки.
2. **Given** проект с Turbo UI, **When** обновляют только tokens (цвета, spacing), **Then** UI обновляется глобально без правки компонентов.

---

### User Story 2 — Разработчик использует компоненты Turbo UI по спецификации (Priority: P1)

Там, где в проекте применяются компоненты Turbo UI, разработчик использует только компоненты библиотеки и токены; не создаёт собственных кнопок, инпутов, карточек вместо компонентов Turbo UI. Блоки по кастомным дизайнам (не из Turbo UI) могут быть сверстаны отдельно.

**Why this priority**: Гарантия стабильности и повторяемости там, где задействована библиотека.

**Independent Test**: Ревью кода компонентов Turbo UI и их использования не находит raw HTML вместо компонентов и hardcoded значений в рамках библиотеки.

**Acceptance Scenarios**:

1. **Given** в экране используются кнопки/инпуты/карточки из Turbo UI, **When** разработчик реализует UI, **Then** применяются только компоненты из Turbo UI и токены.
2. **Given** нужного компонента Turbo UI нет, **When** разработчик расширяет систему, **Then** новый компонент добавляется в Turbo UI и затем используется в проекте.
3. **Given** в проекте есть блок по кастомному дизайну (не из Turbo UI), **When** блок сверстан отдельно, **Then** к нему не предъявляются требования данной спецификации.

---

### User Story 3 — Дизайн и код согласованы через Figma и tokens (Priority: P2)

Дизайн в Figma и реализация в коде согласованы: Figma — reference, реализация — через tokens и компоненты.

**Why this priority**: Снижение расхождений между макетами и реализацией.

**Independent Test**: Сравнение ключевых экранов в Figma и в Storybook; стили в коде ссылаются на токены, а не на произвольные значения.

**Acceptance Scenarios**:

1. **Given** макет в Figma, **When** реализуют экран, **Then** используются токены и компоненты, без копирования «пиксельных» значений из Figma.
2. **Given** изменение дизайна в Figma, **When** обновляют токены или компоненты, **Then** проекты, использующие Turbo UI, получают обновления через обновление библиотеки/токенов.

---

### User Story 4 — Контроль качества и производительности (Priority: P2)

Изменения в Turbo UI не вносят регрессий по производительности и не нарушают принципы конституции.

**Why this priority**: Долгосрочная поддержка и доверие к библиотеке.

**Independent Test**: CI проверяет отсутствие hardcoded значений и inline styles; при необходимости — метрики bundle size и рендера.

**Acceptance Scenarios**:

1. **Given** PR в Turbo UI, **When** в коде есть hardcoded цвет или inline style, **Then** CI не проходит, мерж блокируется.
2. **Given** PR увеличивает bundle size или ухудшает метрики, **When** изменение не обосновано, **Then** PR не мержится до оптимизации или пересмотра.

---

### Edge Cases

- Что делать, если в Figma появился стиль, для которого нет токена? — Добавить токен в `tokens.json`, затем использовать в компоненте; не вводить значение напрямую в коде.
- Как поступать с legacy-проектом, где уже есть hardcoded значения? — Рефакторинг по приоритету; новые участки кода MUST следовать конституции; старые — помечать тикетами на рефакторинг.
- Допустимы ли inline styles для динамических значений (например, width в %)? — Только если значение не относится к дизайн-решению (цвет, шрифт, отступ, radius). Дизайн-решения — только через токены и компоненты.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST хранить все визуальные и типографические решения в `tokens.json` (Single Source of Truth).
- **FR-002**: Система MUST запрещать hardcoded цвета, spacing, размеры шрифтов, border-radius в коде и CSS; допустимы только ссылки на токены или API компонентов.
- **FR-003**: Реализация UI MUST выполняться через компоненты Turbo UI (component-first); запрещены «разовые» raw HTML-элементы с кастомными стилями вместо компонентов.
- **FR-004**: Там, где в проекте используются компоненты Turbo UI, MUST применяться только существующие компоненты библиотеки; создание одноразовых компонентов-заменителей вместо компонентов Turbo UI запрещено. Блоки по кастомным дизайнам (вне Turbo UI) могут быть реализованы отдельно и не подпадают под FR-004.
- **FR-005**: Figma MUST использоваться как reference; реализация в коде MUST следовать tokens и компонентам, без копирования стилей/разметки из Figma в обход токенов.
- **FR-006**: Публичный API компонентов и библиотеки MUST быть минимально необходимым; избыточные props и дублирующие способы достижения одного результата не допускаются.
- **FR-007**: Система MUST иметь единую, предсказуемую стратегию fallback для состояний (loading, error, empty); произвольные или противоречащие fallback запрещены.
- **FR-008**: Изменения, влияющие на bundle size, runtime или рендер, MUST быть обоснованы; регрессии по производительности устраняются до мержа.
- **FR-009**: Production-код MUST быть чистым: без отладочного кода, закомментированных блоков «на потом», временных обходов без тикета на рефакторинг.
- **FR-010**: Версионирование MUST следовать semantic versioning (MAJOR.MINOR.PATCH); версия в package.json и тегах соответствует реальным изменениям.
- **FR-011**: Расширение Design System (новые компоненты, токены) MUST следовать единым правилам: документация/Storybook, использование токенов, соответствие принципам; breaking changes — semantic versioning и changelog.
- **FR-012**: Inline styles MUST быть запрещены для дизайн-решений; нарушения устраняются рефакторингом (стили через токены и компоненты).

### Key Entities

- **Token**: именованное значение дизайн-решения (цвет, spacing, typography, radius и т.д.), хранится в `tokens.json`, используется в коде через переменные темы/утилиты.
- **Component**: переиспользуемый UI-элемент библиотеки Turbo UI; имеет минимальный публичный API, использует только токены для стилей.
- **Figma Library**: визуальный и структурный reference; не источник кода; соответствие через токены и компоненты.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Все визуальные решения в коде Turbo UI и в проектах, использующих Turbo UI, ссылаются на tokens; проверка (ручная или автоматическая) не находит hardcoded цветов/spacing/типографики/radius в production-коде.
- **SC-002**: В компонентах Turbo UI и при их использовании не применяются raw HTML-элементы с кастомными стилями вместо компонентов библиотеки; кастомные блоки проектов (вне Turbo UI) могут быть сверстаны по своим правилам.
- **SC-003**: Inline styles отсутствуют для дизайн-решений; CI или ревью выявляют нарушения до мержа.
- **SC-004**: Новые проекты подключают Turbo UI и переопределяют только tokens для смены стилей; функционал компонентов не меняется при смене темы.
- **SC-005**: Версии Turbo UI в package.json и тегах соответствуют semantic versioning; breaking changes сопровождаются MAJOR bump и записью в changelog.

## Assumptions

- Репозиторий Turbo UI и Figma-файл доступны команде; процесс синхронизации дизайн–код определяется отдельно (ручной ревью или инструменты).
- «Любой проект» означает проекты в экосистеме агентства с возможностью подключения npm-пакета или монорепозитория.
- CI и линтеры могут быть расширены правилами на отсутствие hardcoded значений и inline styles; точный набор правил выносится в реализацию.
