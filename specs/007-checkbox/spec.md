# Feature Specification: Checkbox

**Feature Branch**: `007-checkbox`  
**Created**: 2026-04-17  
**Status**: Draft  
**Input**: Новый компонент дизайн-системы **Checkbox**: по конституции Turbo UI и токенам проекта; документация в Storybook по шаблону библиотеки, текст — коротко, по делу, полезно разработчикам и дизайну (ясный деловой стиль). Размеры **large**, **medium**, **small**. Состояния взаимодействия и отображения: **default**, **hover**, **disabled**, **invalid**. Значения выбора: **включён (true)**, **выключен (false)**, **неопределён / uncertain (indeterminate)**.

**Референс макета (Figma)**: [Turbo UI — node 761:112](https://www.figma.com/design/pj5aiXE1X40rEoVbtyVQ2F/Turbo-UI?node-id=761-112)

## Purpose

**Checkbox** — стандартный переключатель «да/нет» (и режим «частично выбрано» для иерархий и групп). Нужен в формах, таблицах, фильтрах и списках. Визуал и поведение MUST совпадать с макетом и с остальными контролами Turbo UI; оформление только через **design tokens** и правила конституции (без произвольных цветов и отступов вне системы).

## Scope

- Один публичный компонент **Checkbox** на базе нативной семантики флажка (`input type="checkbox"`), с кастомным визуалом по макету (детали API — в плане и контракте).
- **Размеры**: **large**, **medium**, **small** — как на макете; размеры кликабельной зоны, иконки/индикатора и обводки согласованы с сеткой токенов.
- **Состояния взаимодействия и отображения** (визуал по макету): **default**, **hover**, **focus** (видимое кольцо/контур согласно системе), **disabled**, **invalid** (ошибка поля или группы — визуал invalid на макете).
- **Значения выбора**: **включён**, **выключен**, **неопределён (indeterminate)** — поведение и отображение «третьего» состояния как на макете и в нативной семантике (неопределённость не должна путаться с «выключен» визуально).
- **Стили**: только токены темы и принятая в репозитории модель стилизации; Figma — reference, не источник сырых значений в коде.
- **Документация**: Storybook — матрица размеров × состояний × значений, примеры invalid и disabled; страница Docs по **тому же шаблону**, что у Button / TextArea (якорное меню, Figma/GitHub, блоки с примерами). Тексты — кратко, без воды: что делает контрол, какие пропсы за что отвечают, что смотреть дизайнеру.
- **Экспорт**: отдельный entry для потребителей библиотеки, типы и tree-shakeable экспорт, без ломки существующих публичных API.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик подключает Checkbox в продукте (P1)

Разработчик импортирует Checkbox, задаёт выбранное/неопределённое состояние, размер, при необходимости — disabled и invalid. Клик и клавиатура переключают значение согласно правилам доступности; неопределённость можно выставить и снять так, как заложено в API (план/контракт).

**Why this priority**: Без сценария использования в интерфейсе компонент не даёт ценности.

**Independent Test**: Storybook или тестовый экран: рендер всех трёх размеров; переключение checked/unchecked/indeterminate; disabled блокирует смену; invalid отображается по макету.

**Acceptance Scenarios**:

1. **Given** подключённый пакет Turbo UI, **When** рендерится Checkbox в состоянии «выключен», **Then** отображается визуал по размеру и макету.
2. **Given** Checkbox в состоянии «включён», **When** пользователь активирует контрол (мышь или клавиатура), **Then** значение меняется на «выключен» (если не disabled и не запрещено логикой indeterminate).
3. **Given** передан режим **неопределён**, **When** отображается компонент, **Then** виден визуал третьего состояния по макету и не смешивается с «выключен».
4. **Given** **disabled**, **When** пользователь пытается переключить, **Then** значение не меняется, визуал disabled по макету.
5. **Given** **invalid**, **When** отображается компонент, **Then** визуал ошибки соответствует макету; у контрола выставлены согласованные с остальной системой признаки доступности для ошибки (уточнение в плане при необходимости).

---

### User Story 2 — Визуальное соответствие макету и конституции (P1)

Для каждой комбинации размера, интерактивного состояния и значения (checked/unchecked/indeterminate) внешний вид согласован с Figma (node выше). Цвета, обводки, фокус и отступы выражены через семантические токены.

**Why this priority**: Иначе компонент не усиливает единую дизайн-систему.

**Independent Test**: Сравнение статичных кадров в Storybook со скриншотом макета по матрице (размер × состояние × значение).

**Acceptance Scenarios**:

1. **Given** тема по умолчанию, **When** просматриваются large / medium / small для ключевых состояний, **Then** нет систематических расхождений с макетом по контрасту, обводкам и размерам индикатора.
2. **Given** ревью на соответствие конституции, **When** проверяются стили, **Then** нет дублирующих «сырых» значений там, где есть токены.

---

### User Story 3 — Документация в Storybook (P2)

В Storybook доступна страница Checkbox: матрица примеров, invalid и disabled, краткое описание пропсов и сценариев — в том же подходе, что у других компонентов; формулировки короткие и практичные для разработки и дизайна.

**Why this priority**: Снижает время на поиск API и расхождений с макетом.

**Independent Test**: Открыть Docs Checkbox и убедиться в наличии перечисленного и понятных заголовков секций.

**Acceptance Scenarios**:

1. **Given** запущенный Storybook, **When** открыта документация Checkbox, **Then** видны размеры, состояния, три значения выбора и примеры ошибки/блокировки.
2. **Given** читатель документации, **When** читает описание, **Then** понятно, как задать размер, значение, disabled и invalid без чтения исходников.

---

### Edge Cases

- Переход из **indeterminate** в checked/unchecked при клике — по правилам, согласованным с планом (ожидаемое поведение для «выбрать всех» и подобных сценариев).
- **Invalid** без текста сообщения рядом: визуал контрола invalid не должен подменять понятную ошибку формы; связь с текстом ошибки — через композицию или общий паттерн формы (как у других полей).
- Группа чекбоксов и общая ошибка: эта спека задаёт одиночный Checkbox; групповая семантика и лейблы — композиция на стороне продукта или отдельная фича.
- Клавиатура и фокус: фокус виден; переключение с клавиатуры не ломает ожидания пользователя assistive technologies (детали — в плане a11y).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-1**: Система MUST предоставлять компонент **Checkbox** для бинарного выбора с поддержкой состояния **неопределён (indeterminate)**.
- **FR-2**: Checkbox MUST поддерживать размеры **large**, **medium** и **small** в соответствии с макетом; геометрия и типографика (если применима к подписи внутри компонента) MUST опираться на токены.
- **FR-3**: Checkbox MUST поддерживать визуальные и интерактивные состояния: default, hover, focus, disabled и invalid; цвета и обводки MUST задаваться семантическими токенами темы.
- **FR-4**: Для значений **включён**, **выключен** и **неопределён** MUST быть различимый визуал по макету.
- **FR-5**: Стили MUST задаваться только через токены проекта и принятую модель стилей библиотеки; запрещены дублирующие сырые значения для решений, уже выраженных токенами (конституция).
- **FR-6**: Публичный API MUST оставаться **минимально достаточным**; детали пропсов — в плане и контракте.
- **FR-7**: Checkbox MUST быть **согласован** с остальными контролами Turbo UI по фокусу, disabled и invalid там, где это применимо.
- **FR-8**: Компонент MUST экспортироваться отдельным entry; существующие публичные API других компонентов не MUST ломаться.
- **FR-9**: Документация в Storybook MUST включать представление размеров, состояний и значений выбора; тексты MUST быть краткими и полезными для разработчиков и дизайнеров (без лишней воды).
- **FR-10**: Поведение **disabled** и **invalid** MUST соответствовать макету и не вводить пользователя в заблуждение относительно доступности действия.

### Assumptions

- При нехватке токенов для чекбокса (отступы размеров, цвета indeterminate/invalid) токены дополняются по процессу дизайн-системы, а не разовыми константами в компоненте.
- Подпись к чекбоксу (текст справа/слева) в этой фиче может быть **композицией** с типографикой темы или уточняется в плане; отдельный составной `CheckboxField` не обязателен, если макет node 761:112 задаёт только сам контрол.
- «Uncertain» в запросе трактуется как нативное **indeterminate** для групп и частичного выбора.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-1**: Для каждого размера (**large**, **medium**, **small**) и согласованных с макетом состояний (default, hover, focus, disabled, invalid) и значений (включён / выключен / неопределён) визуальное соответствие можно проверить по чек-листу сопоставления со скриншотом макета без расхождений, влияющих на читаемость и иерархию.
- **SC-2**: Разработчик может за **менее чем 15 минут** подключить Checkbox в тестовом проекте и воспроизвести сценарии: переключение значения, неопределённость, disabled и invalid — без правки кода библиотеки.
- **SC-3**: В Storybook доступны примеры, покрывающие три размера, ключевые состояния и три значения выбора; описание на странице Docs позволяет понять назначение пропсов без чтения реализации.
- **SC-4**: Ревью по конституции: для визуальных решений компонента не используются hardcoded значения там, где для них есть токены; нарушения устраняются до принятия фичи.
